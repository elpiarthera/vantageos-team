## What is Browserbase?

Browserbase is managed headless browser infrastructure. It runs Chrome in the cloud so you never maintain your own Chromium fleet. You connect via standard automation frameworks — Playwright, Puppeteer, Selenium — or through their own AI-native SDK called [Stagehand](https://github.com/browserbase/stagehand), and it handles everything else: scaling, CAPTCHA solving, proxy rotation, session replay, and bot-detection bypass.

The company owns and maintains Stagehand as an open-source project. Stagehand sits on top of Playwright and adds AI-driven `act`, `extract`, and `observe` calls. Instead of writing brittle CSS selectors, agents write natural-language instructions. When the UI changes, the AI adapts. No selector maintenance, no flaky tests breaking on every deploy.

Browserbase actually ships three products: the core managed Chrome infrastructure (the engine), Stagehand (the AI-native automation interface), and Director — a visual, no-code workflow builder currently in limited availability.

## Why it matters

The gap between what AI agents can do and what they can see is one of the most overlooked bottlenecks in modern development. Agents can write code, analyze data, generate reports — but they cannot navigate a browser, verify a UI, or interact with web applications the way a human does. Browserbase closes that gap.

For developers building SaaS products, this means CI pipelines that actually test your application on real cloud browsers. For founders building agent systems, it means agents that can scrape JS-rendered pages, fill out forms, navigate login walls, and verify visual output — all without maintaining local Chromium installations or fighting anti-bot detection.

The pricing makes it accessible: the Developer plan at $20/month gives you 100 browser hours and 25 concurrent sessions. That covers most CI workloads comfortably.

## What you can use it for

- **Automated E2E testing in CI**: Connect Playwright to Browserbase via a WebSocket endpoint. Your existing spec files run unchanged — they just execute on cloud Chrome instead of local Chromium. Every PR gets tested automatically without infrastructure overhead.

- **AI-driven browser automation**: Use Stagehand's `act()` and `extract()` calls to navigate complex web flows in natural language. Particularly useful for multi-step wizards, authenticated dashboards, and any UI that changes frequently.

- **Web scraping with JS rendering**: Browserbase's Fetch API handles static content cheaply, with automatic fallback to full browser sessions for JavaScript-heavy pages. The Smart Fetch Scraper pattern — try Fetch first, escalate to full browser only when needed — minimizes cost while handling the full range of web content.

- **Workflow automation on legacy portals**: Persistent browser sessions stay logged in across runs. Stagehand adapts when the vendor updates their portal UI. This is particularly valuable for healthcare, legal, and government portals that lack APIs.

- **Competitive intelligence gathering**: Monitor competitor pricing pages, job postings, and product features on JS-rendered sites. Stealth mode handles bot detection on protected platforms.

## Key takeaways

### 1. The Playwright integration is zero-friction

The most impressive aspect of Browserbase's architecture is how little it demands from existing codebases. If you already have Playwright tests, connecting to Browserbase is a single config change:

```typescript
// playwright.config.ts
use: {
  connectOptions: process.env.BROWSERBASE_API_KEY ? {
    wsEndpoint: `wss://connect.browserbase.com?apiKey=${process.env.BROWSERBASE_API_KEY}`,
  } : undefined,
}
```

When `BROWSERBASE_API_KEY` is set, tests run on cloud Chrome. When it's absent, they run locally. Zero code changes in your spec files. This is the correct way to design an integration — opt-in by environment, invisible to test authors.

### 2. Stagehand changes the economics of test maintenance

Traditional Playwright tests break every time the UI changes. A renamed CSS class, a restructured form, a redesigned navigation — each requires manual selector updates across your test suite. Stagehand's natural-language instructions survive these changes because the AI re-interprets the page on every run.

The tradeoff is speed and cost: each Stagehand action involves an LLM call, making tests slower and adding API costs. The practical strategy is to use standard Playwright for stable, public-facing pages and Stagehand for dynamic, frequently-changing flows. Both can coexist in the same project.

### 3. The Smart Fetch Scraper pattern is the right architecture for scraping

Browserbase's template library includes a pattern worth adopting broadly: try the lightweight Fetch API first (fast, cheap), fall back to a full browser session only when Fetch fails (JS-rendered content, login walls, CAPTCHAs). This tiered approach keeps costs low — the Fetch API is roughly 10x cheaper than full browser sessions — while ensuring you can handle any page on the web.

This replaces the common pattern of running everything through a full headless browser. Most pages don't need it. The fallback catches the ones that do.

### 4. Session persistence unlocks workflow automation

One capability that separates Browserbase from simpler scraping tools is persistent sessions. A browser session can stay logged in across multiple runs, which means agents can interact with authenticated portals over time without re-authenticating on every execution.

This is the foundation for automating legacy portal workflows — data extraction from healthcare systems, government form submission, SaaS-to-SaaS data sync where no API exists. The value proposition is significant: manual portal work that takes days can be automated into minutes.

### 5. The pricing tiers gate important capabilities

Not all Browserbase features are available on all plans. Basic Stealth mode (bot-detection bypass) requires the Developer plan or above. Advanced Stealth — needed for platforms like LinkedIn that use Cloudflare or Datadome — is Scale plan only (custom pricing). Proxy bandwidth is metered separately. If your use case depends on stealth or high-volume scraping, factor in the real tier you'll need, not just the base price.

The Developer plan at $20/month covers most CI and light automation workloads. The Startup plan at $99/month becomes necessary when browser hours exceed 100/month or you need higher concurrency for production agent workloads.

## What we reused in our stack

We integrated Browserbase into our E2E testing pipeline for VantageStarter, our SaaS boilerplate. The Playwright configuration uses the environment-variable switch pattern — `BROWSERBASE_API_KEY` present means cloud execution, absent means local. This lets developers run tests locally during development and automatically switch to cloud Chrome in CI.

We also adopted the Smart Fetch Scraper pattern for our research agents. Instead of routing all web requests through a full browser, agents try lightweight fetching first and escalate to Browserbase only when the content requires JavaScript rendering or authentication. This cut our browser hour usage significantly while maintaining full coverage.

The Stagehand SDK is in our dependencies for testing dynamic, AI-generated UI flows where traditional selectors would break on every iteration.

## The bottom line

Browserbase solves a real infrastructure problem that most teams work around rather than address directly. If you run Playwright tests, the cloud execution alone justifies the Developer plan. If you build agents that need to interact with the web — scraping, monitoring, form automation — the Stagehand SDK and persistent sessions make Browserbase the most practical option available.

Worth exploring for any team running CI with browser tests, building AI agents that need web access, or automating workflows on legacy portals without APIs. Skip it if your testing is unit-only or your scraping targets are all static HTML.
