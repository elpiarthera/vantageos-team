## Key takeaways

### 1. Skills have three layers -- and most people only build one

The presenter breaks down skill anatomy into three sections. Layer one: metadata -- the name (slug), description, and trigger conditions that tell Claude when to activate the skill. Layer two: core instruction -- the objective, workflow steps, constraints, rules, and function triggers. Layer three: resources -- references (PDFs, JSON, guides), templates (example outputs, email formats), activities (interactive prompts requiring human input), assets (images, logos, non-text files), and scripts (deterministic code for calculations). Most tutorials only cover the core instruction. Enterprise skills need all three.

### 2. Never let Claude auto-generate enterprise skills

The presenter is blunt: using Skill Creator to generate skills is what you see on YouTube, and it is what you should not do. When Claude generates its own instructions, you control none of the steps. You do not know what was created, how it will behave under edge cases, or whether it follows your business process. For enterprise clients, every step of a skill must be deliberately authored by someone who understands both the domain and prompt engineering.

### 3. SOP-based skills are non-negotiable for enterprise

LLMs produce probabilistic outputs. For low-risk tasks (LinkedIn posts, content summaries), that variability is acceptable. For accounting, legal documents, compliance reporting, or anything with financial consequences, it is not. The solution: embed Standard Operating Procedures directly into skill instructions. Define every step, every constraint, every expected format. Make the skill deterministic by design, not by luck.

### 4. HITL (Human In The Loop) for high-risk domains

When a skill operates in a high-risk domain -- legal, finance, data security -- it must include checkpoints where a human reviews and approves before the skill continues. The presenter calls these "HITL loops" and positions them as a structural requirement, not an optional safety feature. The skill pauses, presents its work, waits for approval, then proceeds.

### 5. The build-and-iterate debugging framework

When a skill misbehaves, the diagnosis is systematic. Bad outputs? Missing examples -- add few-shot references. Wrong behavior? Missing rules -- add constraints. Wrong MCP tool usage? Instructions are too long and do not target the specific MCP function name. The fix is always to add what is missing, never to rewrite from scratch.

### 6. Skills are a routing system, not a single prompt

The presenter frames skills as routers: when Claude encounters a trigger, it determines which resources it needs (references, templates, scripts), loads them, and follows the workflow. This is fundamentally different from a flat prompt. A skill is a directory structure with an instruction file that orchestrates access to supporting documents. Think of it as a mini-application, not a paragraph of text.
