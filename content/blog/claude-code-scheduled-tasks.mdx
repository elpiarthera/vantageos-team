## Key takeaways

### 1. Loops are session-bound, scheduled tasks are persistent

This is the core distinction. A loop runs inside your current Claude Code session using cron jobs. It fires at your specified interval, but if you close the terminal or the session, the loop dies. Scheduled tasks persist across sessions, support catch-up (they execute missed runs when you reopen), and run indefinitely. The decision framework is simple: need help right now on a project? Use a loop. Need something daily or weekly? Use scheduled tasks.

### 2. Loops auto-expire after 3 days

Every loop has a built-in 3-day timer. Day one, day two, into day three -- then it auto-deletes. This is a safety mechanism to prevent forgotten loops from burning tokens indefinitely. If you need longer than 3 days, either recreate the loop or convert it into a proper scheduled task.

### 3. Context bloat is the real danger

Each loop iteration adds tokens to the session. If you set a loop to fire every 10 minutes with a substantial report, you will hit context limits fast. The presenter flags this as "context rot" -- the session accumulates so much history that the model loses track of earlier context. Design your loop prompts to be lightweight. Request summaries, not full reports.

### 4. Loops compose with skills

You can loop skills directly: `/loop every 20 min run review-PR skill`. The loop fires, invokes the skill, waits the interval, fires again. This composability means you can build sophisticated recurring workflows without writing any additional code -- just chain a loop interval with an existing skill.

### 5. Natural language invocation works

You do not need slash commands. Saying "every 10 minutes, check my ClickUp for new developments" creates the cron job the same way `/loop` does. The underlying tools are `cron create`, `cron list`, and `cron delete`. Claude figures out which to use from context. One-time reminders work too: "at 3 PM, remind me to review the PR" creates a non-recurring cron that self-deletes after firing.

### 6. Loops are per-session and isolated

Two different sessions cannot see each other's loops. If you set a reminder in tab one and a recurring check in tab two, neither session knows about the other's scheduled jobs. This is by design but can surprise you if you expect a unified task queue.

For longer-running processes -- daily email monitoring for leads, weekly report generation -- we use scheduled tasks. The distinction also shaped how we teach automation in our training: loops are for tactical, in-the-moment monitoring; scheduled tasks are for operational, recurring processes.
